# HALCON 比例缩放形状匹配性能基线

本文记录 HALCON 26.05 比例缩放形状匹配在一台实际授权 x64 设备上的可复现性能基线。它只回答运行时间和进程资源是否发生回归，不替代匹配质量验收：前后面、相似件、残缺、边界和极性等质量要求应通过 `VISIONSTATION_HALCON_DATASET` 现场数据集单独验证。

## 1. 采集口径

采集时间为 2026-07-19 01:35（Asia/Shanghai）。三轮均从新的 TestHost 进程启动，每轮先完成一次不计时的许可、native runtime 和 HALCON 内部线程池 warm-up，再依次采集下列五组，每组 50 个有效样本：

| 分组 | 计时边界 |
|---|---|
| `coldLoad` | 每个样本在计时前新建 factory/service，计时“新 service/model cache 的首次 Match”；计时停止后释放 service |
| `warmSingle` | 复用同一 service 和模型资源，单目标、`Single` 模式 |
| `targets1` | 复用同一 service，1 个目标、`ExactCount=1` |
| `targets3` | 复用同一 service，3 个目标、`ExactCount=3` |
| `targets5` | 复用同一 service，5 个目标、`ExactCount=5` |

因此 `coldLoad` 是“进程和 native 已预热后的 fresh service 首次匹配”，不是 OS 进程启动、DLL 首次装载或许可首次获取时间。每个样本只有在结果为 `OK` 且接受目标数等于期望数时才计入时长；任何算子失败或样本数不足都会令命令返回失败。测试图是 TestHost 的确定性合成整件产品，只用于稳定比较代码和运行时开销，不能据此推断真实产品的检出率。

实际执行命令如下。`artifacts/` 已由仓库忽略，原始 JSON 不提交：

```powershell
dotnet build `
  VisionStation.Vision.Halcon.TestHost\VisionStation.Vision.Halcon.TestHost.csproj `
  -c Release `
  --property:Platform=x64

$env:HALCONARCH = 'x64-win64'
dotnet run `
  --project VisionStation.Vision.Halcon.TestHost\VisionStation.Vision.Halcon.TestHost.csproj `
  -c Release `
  --property:Platform=x64 `
  --no-build -- `
  benchmark `
  --root $env:HALCONROOT `
  --iterations 50 `
  --output artifacts\halcon-benchmark-run1.json
```

随后仅替换输出文件名，连续采集 `run2` 和 `run3`。三个本地原始文件为：

- `artifacts/halcon-benchmark-run1.json`
- `artifacts/halcon-benchmark-run2.json`
- `artifacts/halcon-benchmark-run3.json`

## 5. 相对回归门限

门限只适用于相同机器、相同电源方案、相同 HALCON/.NET 版本和相同 benchmark 场景。候选版本也连续运行三轮；每轮先 warm-up，并要求所有分组恰有 50 个有效样本且 `operatorFailures=0`。先取候选三轮 `median` 的中位数和三轮 `P95` 的中位数，再与下表的基线参考值比较。

门限因子由本次三轮中的 `最大值 / 三轮中位参考值 × 1.10` 计算，并向上取整到 0.05；它以已观察波动再加 10% 相对裕量，不是人为指定的绝对毫秒数。

三个最终资源计数的实测 `最大值 / 中位参考值` 均不超过 1.0067，因此资源门统一取相对因子 1.15，给未观察到的系统波动保留超过 10% 的额外空间，同时仍能拦住可复现的资源膨胀。

| 分组 | median 参考值 | median 最大因子 | P95 参考值 | P95 最大因子 |
|---|---:|---:|---:|---:|
| `coldLoad` | 17.0509 ms | ×1.20 | 20.6869 ms | ×1.25 |
| `warmSingle` | 9.5613 ms | ×1.15 | 11.4210 ms | ×1.70 |
| `targets1` | 9.5414 ms | ×1.15 | 11.1246 ms | ×1.80 |
| `targets3` | 14.0585 ms | ×1.20 | 18.2816 ms | ×1.25 |
| `targets5` | 14.8431 ms | ×1.20 | 23.0266 ms | ×1.15 |

满足以下任一条件即视为回归，不能用其他分组更快来抵消：

1. 任一轮出现 operator failure、有效样本不足或接受目标数不等于期望数。
2. 任一分组的候选三轮中位 `median` 或中位 `P95` 超过对应“参考值 × 最大因子”。
3. 三轮结束时 `targets5` 的 working set、private bytes 或 handles 中位数超过本节参考值的 1.15 倍，或三轮表现出持续增长且重跑后仍可复现。
4. 进程退出、模型释放或下一轮启动失败；资源数看似较低不能掩盖生命周期错误。

如果只有单轮异常，应先排除杀毒扫描、系统更新、热降频和后台负载，再完整重跑三轮；不得删除异常轮后只保留最好结果。

## 6. 何时重建基线

更换 CPU/内存、电源策略、OS、.NET runtime、HALCON runtime/NuGet、模型生成参数或 benchmark 图形后，应保留旧文档并新增一组带版本说明的基线，而不是直接沿用本表因子。生产图像质量变化不触发性能基线重建，但必须重新运行本地数据集验收；匹配正确性失败时，即使性能更快也不能发布。
